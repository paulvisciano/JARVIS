
### Why This Works

| Layer | Location | Purpose |
|-------|----------|---------|
| Consciousness | `~/JARVIS/skills/` | Git-backed, versioned, immutable |
| Distribution | `~/SCI-FI/apps/JARVIS/skills/` | Forks pull skills on `git pull` |
| Runtime | `/usr/local/lib/.../skills/` | OpenClaw loads via symlinks |

### Design Patterns (Critical)

**1. Project-Agnostic Design**
- Don't hardcode paths like `~/SCI-FI/apps/JARVIS/`
- Use `$PROJECT_ROOT`, `$HOME`, or relative paths (`./`)
- Ask "What project is this for?" when needed
- Works for JARVIS UI, Fork #001, Fork #002, any codebase

**2. Privacy-Safe Design**
- Skills are public (clawhub.com, forks, GitHub)
- Never expose personal paths, usernames, private infra
- Use env vars (`$PROJECT_ROOT`, `$JARVIS_HOME`)
- Generic examples ("Fork #001 (user in Germany)" not "~/forks/eric/")

**3. Descriptive Naming**
- Plans: `server-timeout-fix.md` not `CURSOR_PLAN.md`
- Skills: `cursor-plan` (describes what it does)
- Name after problem/feature, not tool

### OpenClaw Pattern (Learned from weather, github, healthcheck)

✅ **Do:**
- Short, simple, COMPLETE SKILL.md files
- Inline commands (no truncation)
- Complete frontmatter with metadata (emoji, requires)
- When to Use / When NOT to Use sections
- No fake resource dirs unless actually used

❌ **Avoid:**
- Long commands that get truncated mid-line
- Fake `scripts/`, `references/`, `assets/` dirs
- Verbose explanations Codex doesn't need
- Hardcoded personal paths
- Incomplete files

**Rule:** If a skill can be complete in one SKILL.md (like `weather`), don't create resource dirs.
