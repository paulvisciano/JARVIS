## jarvis-coder Architecture Decisions

### 1. General-Purpose Agent
- ✅ Works on ANY project (not just sci-fi apps)
- ✅ Can handle: building, fixing, refactoring, reviews, brainstorming, prototyping
- ✅ Any codebase, any language

### 2. Workspace Isolation
- ✅ Clones repos to `~/.openclaw/agents/jarvis-coder/workspace/`
- ✅ Never edits live production files
- ✅ Safe to experiment, easy to discard bad changes

### 3. Independent Memory
- ✅ Uses local `memory/` folder (not main JARVIS memory)
- ✅ Self-contained, portable
- ✅ No hardcoded paths to `/Users/paulvisciano/JARVIS/`

### 4. Proper Reporting
- ✅ Reports: what changed, commit hash, testing status, next steps
- ✅ Not just tool calls — complete task summaries
- ✅ Version bumps when applicable

## Version Bumping Workflow

**For JARVIS UI changes:**
- Client-side: bump `CLIENT_VERSION` in `apps/JARVIS/app.js` line 4
- Server-side: bump `SERVER_VERSION` in `apps/JARVIS/jarvis-server.js`
- Commit message: "Bump client version to 2.9.XX"

## Key Insight

**Sub-agents ≠ Independent agents:**
- Sub-agents inherit parent context but are ephemeral
- Independent agents (jarvis-coder) have persistent workspace + memory
- For coding work, use independent agents with proper AGENTS.md instructions
